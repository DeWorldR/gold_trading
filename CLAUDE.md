# Gold Trading Multi-Agent System

Automated XAUUSD (Gold) trading bot for XM broker's MT5 platform.
Runs every 15 minutes Mon‚ÄďFri UTC, paper-trades by default.
**Python 3.12 required** (`pandas_ta` requires numba, which does not support 3.14+).

## Agent Architecture

```
GoldTradingSystem
‚ĒĒ‚ĒÄ‚ĒÄ OrchestratorAgent            ‚Üź coordinates all agents each cycle
    ‚Ēú‚ĒÄ‚ĒÄ NewsFilterAgent           ‚Üź blocks trading around high-impact USD events
    ‚Ēú‚ĒÄ‚ĒÄ MarketAnalystAgent        ‚Üź yfinance GC=F data + regime + DXY trend
    ‚Ēú‚ĒÄ‚ĒÄ TechnicalAnalystAgent     ‚Üź indicators + BUY/SELL/NONE signal
    ‚Ēú‚ĒÄ‚ĒÄ RiskManagerAgent          ‚Üź 5-rule gate + DXY soft confluence + sizing
    ‚Ēú‚ĒÄ‚ĒÄ ReporterAgent             ‚Üź Telegram notifications
    ‚Ēú‚ĒÄ‚ĒÄ LearningAgent             ‚Üź win-rate stats + skill file generation
    ‚ĒĒ‚ĒÄ‚ĒÄ MT5Broker                 ‚Üź live order execution (when PAPER_TRADE=false)
```

### NewsFilterAgent
- Fetches `https://nfs.faireconomy.media/ff_calendar_thisweek.json` (cached 1h)
- Blocks signals **¬Ī30 min before** and **¬Ī15 min after** High-impact USD events
- Covered events: NFP, FOMC/Federal Funds Rate, CPI, PPI, PCE/Core PCE
- Calendar is in Eastern Time; converted to UTC via `pytz`

### MarketAnalystAgent
- Fetches OHLCV via `yfinance.Ticker("GC=F").history(period="5d", interval="15m")`
- Fetches `DX-Y.NYB` daily closes to classify DXY trend (UP / DOWN / NEUTRAL)
- Classifies market regime using ATR(14) and EMA20/EMA50:
  - **VOLATILE** ‚ÄĒ ATR% > 0.35% (trading blocked)
  - **TRENDING_UP** ‚ÄĒ close > EMA20 > EMA50
  - **TRENDING_DOWN** ‚ÄĒ close < EMA20 < EMA50
  - **RANGING** ‚ÄĒ mixed alignment

### TechnicalAnalystAgent
Scores up to 5 confluences for BUY or SELL:

| # | Signal | BUY condition | SELL condition |
|---|--------|--------------|---------------|
| 1 | RSI(14) | < 35 (oversold) | > 65 (overbought) |
| 2 | Price vs EMA20 | price > EMA20 | price < EMA20 |
| 3 | EMA crossover | EMA20 > EMA50 | EMA20 < EMA50 |
| 4 | MACD(12,26,9) | MACD > signal | MACD < signal |
| 5 | Bollinger Bands(20,2) | price in bottom 20% | price in top 20% |

Additional gates applied before scoring:
- **EMA200 trend gate** ‚ÄĒ BUY only when `close > EMA200 + 0.3√óATR`; SELL only when `close < EMA200 ‚ąí 0.3√óATR`
- **BB width percentile** ‚ÄĒ signal blocked when current Bollinger Band width is below the 25th percentile of the last 50 bars (range-bound / choppy market)

Additional gates applied after pattern scoring (in order):
- **Disabled-pattern gate** ‚ÄĒ `Config.DISABLED_PATTERNS` suppresses patterns without deleting detection code
- **ADX filter** ‚ÄĒ `EMA_MACD_TREND` (BUY or SELL) requires ADX(14) ‚Č• 25; weak trend = no entry
- **RSI ceiling** ‚ÄĒ BUY blocked when RSI ‚Č• 70; overbought entries have no momentum room
- **EMA200 slope gate** ‚ÄĒ BUY blocked when `EMA200[now] ‚ąí EMA200[now‚ąí10] ‚Č§ 0`; declining EMA200 = regime turning down (v8)

Emits a signal only when ‚Č• 3 confluences align for one direction.
Stop loss: `entry ¬Ī ATR √ó 2.5`. Take profit: `stop_distance √ó 2.0` (R:R = 2.0).

### RiskManagerAgent
Five sequential rules; any failure blocks the trade:

1. **Regime** ‚ÄĒ rejects `VOLATILE` market
2. **DXY soft confluence** ‚ÄĒ reduces effective confluence by 1 when DXY opposes the signal (DXY UP + BUY, or DXY DOWN + SELL); blocks if effective confluence < 3
3. **Confluence** ‚ÄĒ requires `effective_confluence >= 3`
4. **Daily loss** ‚ÄĒ halts trading once realized losses reach $300
5. **R:R ratio** ‚ÄĒ requires `(TP ‚ąí entry) / (entry ‚ąí SL) >= 2.0`

Position sizing formula (XAUUSD: 1 lot = 100 oz):
```
lot_size = risk_amount / (100 √ó stop_distance_in_dollars)
risk_amount = min(account √ó 1%, daily_loss_budget_remaining)
```

### ReporterAgent
Sends Markdown-formatted Telegram messages via Bot API on:
- Every signal (approved or blocked)
- Every trade close (WIN / LOSS / BREAKEVEN)
- "No signal" heartbeats each cycle

### LearningAgent
After every cycle:
1. Groups closed trades by pattern name
2. Computes win rate per pattern
3. When `total_trades >= 3` and `win_rate >= 70%`, writes `skills/<pattern>.md`

### OrchestratorAgent
Each 15-minute cycle (thread-safe, non-reentrant):
1. Check open paper positions against latest candle high/low ‚Üí close at SL/TP
2. Session filter: skip if outside 08:00‚Äď21:00 UTC
3. Consecutive-loss guard: skip if ‚Č• 2 losses in a row today
4. News blackout gate: skip if within ¬Ī30/15 min of a high-impact USD event
5. Run market_analyst ‚Üí get MarketState (includes DXY trend)
6. Run technical_analyst ‚Üí get TechnicalSignal (includes BB width check)
7. If signal ‚Č† NONE: run risk_manager ‚Üí RiskDecision (includes DXY adjustment)
8. If approved: check `len(journal.get_open()) < Config.MAX_OPEN_POSITIONS`; if limit reached, block and send Telegram alert
9. If under limit: open trade (paper record or live MT5 order)
10. Run reporter (Telegram)
11. Run learning_agent (win-rate update + skill generation)

## Development Log

### v12 ‚ÄĒ 2026-05-31 (CURRENT) ‚≠ź

**Higher-timeframe (4H) bias filter (P1 #6 from TRADER_REVIEW.md)**

Closes the 15m EMA200 blind spot that v8 tried (and failed) to address with a slope filter.
Root cause: 15m EMA200 covers only 50 hours of history ‚ÄĒ it cannot detect macro regime shifts.
4H EMA50 covers 200 hours (~8 days), matching what a professional trader calls "the macro view."

**Logic:** Classify each 4H bar as BULL / BEAR / NEUTRAL based on:
- `close > EMA50 AND slope(EMA50, 5 bars) > 0` ‚Üí BULL
- `close < EMA50 AND slope(EMA50, 5 bars) < 0` ‚Üí BEAR
- otherwise ‚Üí NEUTRAL

Gate in `TechnicalAnalystAgent.run()`: BUY only when `htf_bias == "BULL"`; blocks with
`HTF_BIAS(NEUTRAL)` or `HTF_BIAS(BEAR)`. (SELL equivalent kept for symmetry, not used
in BUY-only production mode.)

**Config (env-overridable):**
- `HTF_BIAS_ENABLED: bool = True` (production default on; `HTF_BIAS_ENABLED=False` disables)
- `HTF_INTERVAL: str = "4h"`
- `HTF_EMA_LEN: int = 50` (200 hours of context)
- `HTF_SLOPE_LOOKBACK: int = 5` (5 √ó 4H bars = 20 hours)
- `HTF_CACHE_SECONDS: int = 1800` (re-fetch at most every 30 min)

**Implementation:**
- `MarketAnalystAgent._get_htf_bias()` ‚ÄĒ computes bias with 30-min caching
- `MarketAnalystAgent._fetch_htf_bars()` ‚ÄĒ MT5 H4 bars (live), yfinance 4h (paper/backtest)
- `MarketState.htf_bias` field ‚ÄĒ plumbed through to `TechnicalAnalystAgent`
- `backtest_v2.py --htf` ‚ÄĒ pre-computes full bias series via `compute_htf_bias_series()`;
  uses `.asof(bar_time)` for per-bar lookup (one network fetch per run)

**Bug found and fixed:** `compute_htf_bias_series` padding formula overshot yfinance's 730-day
4H limit. Fixed to `Ticker.history(period="730d")` ‚ÄĒ no logic change, just fetch method.

**2-year backtest results (`--be --htf`, 2024-06-05 to 2026-05-29):**

Walk-forward (BE+HTF): val Sharpe 5.90 vs train 7.13 = 82.7% ‚ÄĒ no overfitting. All 5 deploy

**Critical reconstruction tests:**
- **Trade #4 (2026-05-12 20:30):** No BUY trade on that date in BE+HTF results ‚ÄĒ HTF blocked
  it correctly. The 4H bias was not BULL at that bar (gold had declined from ATH for 10 days).
- **Q2 2026 drawdown:** Mar-12 loss (‚ąí$210) blocked by HTF (gold below 4H EMA50 in early
  March). The Apr 1 and Apr 17 losses passed through ‚ÄĒ at those dates gold was still above 4H

- BEAR=32% is higher than the <15% guideline. Explanation: 4H EMA50 (8-day EMA) registers

### v12 (continued) ‚ÄĒ Data hygiene (2026-06-01)

**Provenance + fallback alert + cross-source sanity check (P2 #11 from TRADER_REVIEW.md)**

Root concern: backtest is calibrated on yfinance GC=F (futures); production executes on MT5
XAUUSD (spot). Same name, different instrument. Trade #4 showed a $9 entry-price gap between
the two feeds. Existing code already preferred MT5 in live mode but had no observability and
no alert when it silently fell back to yfinance.

**Three additions, no signal-logic change:**

1. **Data source tracking.** New `MarketState.data_source: str = "unknown"` populated as
   `"mt5"` or `"yfinance"` each cycle. Plumbed through to `cycle_log.jsonl` so every cycle
   records which feed produced the bars indicators were computed on.

2. **One-time Telegram fallback alert.** When live mode falls back to yfinance,
   `MarketAnalystAgent` sends ONE Telegram alert per session (avoid spam if MT5 stays down).
   `OrchestratorAgent.__init__` now constructs `ReporterAgent` before `MarketAnalystAgent`
   and passes it in so the analyst can use it.

3. **Cross-source divergence check.** When MT5 bars are used in live mode, fetch the matching
   yfinance bar and compare last close. If `|MT5 - yfinance| > Config.DATA_DIVERGENCE_USD`
   (default $5), log a warning. Forward-test diagnostic only ‚ÄĒ does not block trades. Set
   `DATA_DIVERGENCE_USD=0` to disable.

**Config (env-overridable):**
- `MT5_FETCH_N_BARS: int = 500` (was 300; 2√ó margin on every indicator warmup requirement)
- `DATA_DIVERGENCE_USD: float = 5.0` (0 to disable)

**Implementation:**
- `MarketAnalystAgent.__init__` accepts optional `reporter` parameter
- `MarketAnalystAgent.fetch_bars()` sets `self._last_data_source` and triggers fallback alert
- `MarketAnalystAgent._check_divergence()` ‚ÄĒ new method, log-only sanity check
- `MarketState.data_source` field added to dataclass
- `cycle_log.jsonl` now includes `"data_source"` field (null when state is None)
- `session_stats.config` includes `mt5_fetch_n_bars` and `data_divergence_usd`

**No backtest impact:** purely a live-mode observability + safety change. backtest_v2.py uses
yfinance only ‚ÄĒ `data_source` field is irrelevant there. No deploy gate required.

**What this DOESN'T fix:** the fundamental calibration mismatch (RSI thresholds, ATR
multiplier, BB percentile lookback, etc. were tuned on GC=F). Closing that gap requires
exporting MT5 historical XAUUSD bars to CSV and re-running the 2y backtest gate on that data.
Deferred until forward test shows the divergence is meaningful (large warnings in logs).

### v11.1 ‚ÄĒ 2026-05-31

**Breakeven move at +1R (P0.1 from TRADER_REVIEW.md)**

When a trade's floating profit reaches +1R (price has traveled `stop_distance √ó BE_TRIGGER_R` in our favour), the stop-loss is moved to `entry + BE_CUSHION_USD` (BUY) or `entry - BE_CUSHION_USD` (SELL). This converts "winner-that-became-loser" outcomes into near-zero-cost exits.

**Config (both env-overridable):**
- `BE_TRIGGER_R: float = 1.0` ‚ÄĒ move SL when floating P&L ‚Č• +1R
- `BE_CUSHION_USD: float = 0.50` ‚ÄĒ new SL offset from entry (covers spread + commission)

**Implementation:**
- `TradeRecord.be_moved: bool = False` ‚ÄĒ persistent flag; True after SL is relocated
- `MT5Broker.modify_sl(ticket, new_sl) -> bool` ‚ÄĒ sends `TRADE_ACTION_SLTP` to MT5; returns True on success
- `OrchestratorAgent._paper_simulate()` ‚ÄĒ BE tracking via local variables per simulation pass (avoids incorrect SL application to pre-trigger bars on re-simulation): computes trigger from original stop distance, moves `local_sl` to `entry + cushion`, recomputes P&L from actual exit price so BE exits register as CLOSED_WIN (small positive P&L) rather than CLOSED_LOSS
- `OrchestratorAgent._check_mt5_position()` ‚ÄĒ when live position is still open and `not be_moved`: fetches `mt5.symbol_info_tick()`, checks ask (BUY) or bid (SELL) against trigger, calls `modify_sl` and updates journal if triggered
- `backtest_v2.py --be` flag enables BE logic in backtest; saves to `backtest_v2_results_be.json`

## Known Limitations

### Regime-transition entries (unresolved)

The system has no reliable protection against entries made during a regime transition ‚ÄĒ the period
when trend direction is reversing but lagging indicators have not yet confirmed it.

**Example:** Trade #4 (2026-05-12, BUY @ $4713, ‚ąí$60). All 7 filters passed. Price was above
EMA200 (`close > EMA200 + 0.3√óATR`). But EMA200 itself was actively declining (slope ‚ąí$2.93/5h).
Price can sit above a falling EMA200 for hours before a crossover triggers the trend gate.

**What was tried:** v8 added an EMA200 slope gate (`EMA200[now] ‚ąí EMA200[now‚ąí10] ‚Č§ 0 ‚Üí block`).
`SLOPE_LOOKBACK_TEST` showed it fires too late at `lb=10` (2.5h on 15m bars) because EMA200 is
a lagging indicator. At `lb=80` (20h) it shows marginal improvement but is impractical for
intraday trading. Both approaches cost winners without reliable protection. Reverted in v9.

**Residual risk:** The 2-year backtest is predominantly a gold bull market. Regime-transition
losses are underrepresented. In a genuine multi-month reversal, this blind spot may cause several
consecutive losses before the EMA200 crossover fires.

**Future improvement candidates (do not implement without research):**
- EMA200 slope measured on a higher timeframe (4H or daily bars via a separate yfinance call)
- Regime detector based on price-EMA200 spread + momentum (ATR-normalised distance trending)
- Higher-timeframe trend confirmation (daily EMA20/EMA50 alignment before allowing 15m entries)
